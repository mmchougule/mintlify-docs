This page enumerates the privacy properties b402-solana actually provides
on mainnet today, with file:line citations to the on-chain code that
enforces each one. Anything not listed here is not claimed.

## What's hidden

### 1. Note contents (amount, mint, ownership)

Every shielded balance lives in the pool tree as a Poseidon commitment

```
commitmentHash = Poseidon(TAG_COMMIT, mintFr, valueFr, random, spendingPub)
```

The pool stores the leaf, not the preimage. A Groth16 proof binds the
amount, mint, and ownership in zero-knowledge during every spend; the
verifier program never receives the value or the spending key. See the
commitment construction in
[`programs/b402-pool/src/instructions/adapt_execute.rs:939-956`](https://github.com/mmchougule/b402-solana/blob/main/programs/b402-pool/src/instructions/adapt_execute.rs#L939-L956)
(the excess-leaf path under dual-note minting mirrors the SDK's `commitmentHash` byte-for-byte)
and the verifier in
[`programs/b402-verifier-adapt/src/lib.rs:40-52`](https://github.com/mmchougule/b402-solana/blob/main/programs/b402-verifier-adapt/src/lib.rs#L40-L52).

### 2. Sender → recipient unlinkability

When the relayer signs a post-shield op (swap, lend, redeem, perp open),
the trader's wallet is not in the transaction. The recipient binding is
proof-bound as
`Poseidon_3(TAG_RECIPIENT_BIND, owner_low_128, owner_high_128)` and never
exposed as a pubkey on the wire.

Concrete artifact (mainnet): the privateSwap tx
[`5aVtM9Vc3eSgakTwv8vNG2snokQCWZhc8D6vfLTYigJRTkW2eZ8Ymv5gYawD81qCfAyj3KzhvHJqvGA1znpLKUhB`](https://solscan.io/tx/5aVtM9Vc3eSgakTwv8vNG2snokQCWZhc8D6vfLTYigJRTkW2eZ8Ymv5gYawD81qCfAyj3KzhvHJqvGA1znpLKUhB)
is signed by the hosted relayer
`7f6gRiX56dMQGrPERNBKuzFsvagFTM1U4LMAAN9rsiNM`, not by the trader's
wallet. The trader's pubkey appears in zero accounts on this transaction.

### 3. Slab-side wallet isolation (per-protocol PDAs)

Stateful adapters (Kamino today, Drift / Marginfi / Adrena scaffolded)
register positions in the downstream protocol under a PDA derived from the
trader's spending key — never under the trader's wallet.

```rust
// programs/b402-kamino-adapter/src/lib.rs:1899-1908
pub fn derive_owner_pda(
    adapter_program_id: &Pubkey,
    viewing_pub_hash: &[u8; 32],
    mint: &Pubkey,
) -> (Pubkey, u8) {
    Pubkey::find_program_address(
        &[VERSION_PREFIX, SEED_ADAPTER_OWNER, viewing_pub_hash.as_ref(), mint.as_ref()],
        adapter_program_id,
    )
}
```

The protocol's own state (Kamino `Obligation`, eventual Drift `User`,
percolator slab account) records the adapter-derived `owner_pda`, not the
trader's wallet. Two consequences:

1. **Different traders → different PDAs by construction.** The PDA seeds
   include `viewing_pub_hash` which is unique per shielded note holder.
2. **Cross-protocol correlation is impossible from the PDA alone.** The
   PDA is also scoped by `adapter_program_id`, so the same trader on the
   Kamino adapter and the Drift adapter resolves to two unrelated
   addresses — even an observer with both adapters' full state can't
   join them on `owner_pda`.

### 4. Per-user state isolation on mainnet Kamino

Each viewing key gets its own Kamino `Obligation` PDA, mediated by the
adapter's `per_user_obligation` build (see
[`programs/b402-kamino-adapter/src/lib.rs:222-239`](https://github.com/mmchougule/b402-solana/blob/main/programs/b402-kamino-adapter/src/lib.rs#L222-L239)
for the per-user `remaining_accounts` layout that pins `OWNER_PDA` and
`OWNER_USDC_ATA`). The previously-documented "everyone shares one
obligation" risk no longer applies on the shipped mainnet adapter.

The `cpi-only` gate at
[`programs/b402-kamino-adapter/src/lib.rs:353-371`](https://github.com/mmchougule/b402-solana/blob/main/programs/b402-kamino-adapter/src/lib.rs#L353-L371)
is what makes per-user safe: it rejects any `execute()` invocation whose
outer instruction is not the b402 pool program, so an attacker who reads
a depositor's `viewing_pub_hash` off-chain cannot directly call the
adapter's withdraw with that hash to drain the user.

## What's NOT hidden

State each one specifically rather than under an "everything else" cover.

- **`viewing_pub_hash` linkability across the same user's ops.** The
  shielded engine needs `viewing_pub_hash` (= `outSpendingPub[0]` in the
  24-input vector — see
  [`programs/b402-pool/src/instructions/adapt_execute.rs:88-95`](https://github.com/mmchougule/b402-solana/blob/main/programs/b402-pool/src/instructions/adapt_execute.rs#L88-L95))
  to find a user's pending-inputs PDA and to derive their per-user
  protocol PDAs. Two ops by the same trader against the same stateful
  adapter share an `owner_pda`. This is **not a wallet link** — it's a
  pseudonym shared across the user's own positions inside one protocol.
- **Trade size, mint, slippage parameters.** Same exposure as any DEX.
  `public_amount_in`, `expected_out_value`, and the IN/OUT mint Frs are
  on the public-input vector
  ([`adapt_execute.rs:1116-1163`](https://github.com/mmchougule/b402-solana/blob/main/programs/b402-pool/src/instructions/adapt_execute.rs#L1116-L1163)).
- **Aggregate pool TVL by mint.** Each `vault` SPL TokenAccount is a
  PDA-owned canonical account; balances are public.
- **Network-layer metadata.** A relayer sees the user's IP unless the
  user uses Tor. The protocol does not anonymize the network layer.
- **Tx timing.** Small TVL = thin anonymity set. With pool throughput
  measured in tens of ops/day, a user shielding 100 USDC and unshielding
  99.95 USDC two minutes later collapses their own anonymity set to one.
- **Anything the user discloses.** Viewing keys are user-sovereign; if a
  user hands an auditor their viewing key, the auditor sees their notes.
  No on-chain forced-disclosure mechanism.

## Two-artifact framing

The privacy claim decomposes into two properties, each backed by an
on-chain artifact you can fetch and inspect.

### A. Tx-level wallet isolation

**Property.** A post-shield op transaction does not contain the trader's
wallet pubkey in any account slot.

**Artifact.** Mainnet privateSwap tx
[`5aVtM9Vc…sLKUhB`](https://solscan.io/tx/5aVtM9Vc3eSgakTwv8vNG2snokQCWZhc8D6vfLTYigJRTkW2eZ8Ymv5gYawD81qCfAyj3KzhvHJqvGA1znpLKUhB).
Signer: `7f6gRiX56dMQGrPERNBKuzFsvagFTM1U4LMAAN9rsiNM` (hosted relayer).
Trader wallet: not present.

### B. Per-protocol state isolation

**Property.** When a stateful adapter (Kamino, percolator, eventually
Drift) writes per-user state in the downstream protocol, the recorded
"owner" is a PDA derived from the trader's `viewing_pub_hash`, not their
wallet.

**Artifact.** Devnet percolator keystone tx
[`2NwVGzPufiL7W4gneKgSjJNh9fKMnqwKYoL4heNawdWFHmdme6HjhCFzqdYf7i4NUmpCf6v3gUxBsHb4x6Wnb6ZR`](https://solscan.io/tx/2NwVGzPufiL7W4gneKgSjJNh9fKMnqwKYoL4heNawdWFHmdme6HjhCFzqdYf7i4NUmpCf6v3gUxBsHb4x6Wnb6ZR?cluster=devnet).
The percolator slab account records the position owned by the
b402-derived PDA, not by the funding wallet — same code path as the
mainnet kamino artifact above, just with the percolator adapter program
ID instead of the kamino adapter.

Both artifacts exercise the same SDK call (`b402.adaptExecute`) into the
same pool instruction (`adapt_execute_v2`) — only the adapter ID and the
downstream CPI ladder differ.

## Trust assumptions

A short list of things this page assumes; full discussion in
[/concepts/trust-assumptions](/concepts/trust-assumptions).

- **Trusted setup.** The adapt VK in
  [`programs/b402-verifier-adapt/src/vk.rs`](https://github.com/mmchougule/b402-solana/blob/main/programs/b402-verifier-adapt/src/vk.rs)
  is from a single-contributor ceremony today. A multi-party ceremony is
  on the roadmap.
- **Hosted relayer.** Wallet-isolation (property A above) holds because
  the relayer signs the post-shield op. A user who self-submits voids
  property A — their wallet appears as the signer. The SDK currently
  permits self-submission only on devnet for testing; mainnet flows go
  through the hosted relayer.
- **Light Protocol nullifier tree V2.** Nullifier inserts CPI into
  `b402_nullifier::create_nullifier`, which writes into Light V2's
  address tree. See the inline-CPI path in
  [`programs/b402-pool/src/instructions/adapt_execute.rs:609-661`](https://github.com/mmchougule/b402-solana/blob/main/programs/b402-pool/src/instructions/adapt_execute.rs#L609-L661).
  Privacy depends on Light's IMT continuing to admit valid nullifiers
  without leaking the spent leaf index.
