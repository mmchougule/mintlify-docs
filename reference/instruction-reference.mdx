Accounts: relayer, PoolConfig, AdapterRegistry, TokenConfig (in),
TokenConfig (out), Vault (in, mut), Vault (out, mut), TreeState (mut),

NullifierShard[p0] (mut), NullifierShard[p1] (mut), token_program,
system_program.

`remaining_accounts` are forwarded to the adapter CPI.

### Admin instructions

| Instruction | Args |
|---|---|
| `pause(which: u8)` | bitmask: which subsystem to pause |
| `unpause(which: u8)` | bitmask: which subsystem to unpause |
| `set_verifier(target: VerifierTarget, new_program_id: Pubkey)` | swap verifier program for `transact` or `adapt` |
| `register_adapter(info: AdapterInfo)` | append-only allowlist add |
| `add_token_config(args: AddTokenConfigArgs)` | whitelist SPL mint |
| `set_max_tvl(mint: Pubkey, amount: u64)` | update per-mint TVL cap |

There is no on-chain `disable_adapter`, `enable_adapter`,
`queue_upgrade`, `activate_upgrade`, `cancel_upgrade`,
`revoke_upgrade_authority`, or `rotate_admin` instruction. Admin is a
single key today; multi-signer enforcement is reserved.

## Anchor discriminators

First 8 bytes of `sha256("global:<ix_name>")`. Computed at build time
into the program binary. The SDK helper:

```typescript
import { instructionDiscriminator } from '@b402ai/solana';
const disc = instructionDiscriminator('shield');
// = sha256("global:shield").slice(0, 8)
// = [0xdc, 0xc6, 0xfd, 0xf6, 0xe7, 0x54, 0x93, 0x62]
```

## Encoded instruction layouts (wire format)

For low-level inspection / debugging, the wire format of each
instruction is `discriminator || borsh_args`. Borsh in Anchor uses
little-endian primitive encoding, length-prefixed `Vec`, and
struct-fields-in-declaration-order.

### Shield (1,099 bytes typical)

```
[0..8]       discriminator       sha256("global:shield")[..8]
                                 = dc c6 fd f6 e7 54 93 62
[8..12]      u32 LE              proof.len() = 256
[12..268]    proof               Groth16 (256 B)
[268..580]   TransactPublicInputs (312 B)
                merkle_root        [32]
                nullifier          [32][32]
                commitment_out     [32][32]
                public_amount_in   u64 LE
                public_amount_out  u64 LE
                public_token_mint  [32]
                relayer_fee        u64 LE
                relayer_fee_bind   [32]
                root_bind          [32]
                recipient_bind     [32]
[580..584]   u32 LE              encrypted_notes.len()
[584..]      encrypted_notes      [89B ct + 32B ephPub + 2B vt]*
[..]         u8                  note_dummy_mask
```

### Unshield

Same shape as shield with these differences:

- `public_amount_in == 0`, `public_amount_out > 0`
- `recipient_bind` non-zero (committed to recipient ATA owner)
- `nullifier` slots are real (not 0 sentinels) when spending real notes
- Trailing `relayer_fee_recipient: Pubkey` arg

### Adapt_execute (~1,231 B with Jupiter route)

```
[0..8]       discriminator       sha256("global:adapt_execute")[..8]
[8..12]      u32 LE              proof.len() = 256
[12..268]    proof               Groth16
[268..]      AdaptPublicInputs   (376 B — adds 4×32 + 8 over transact)
[..]         u32 LE              encrypted_notes.len()
[..]         encrypted_notes     [89+32+2]*
[..]         u8                  in_dummy_mask
[..]         u8                  out_dummy_mask
[..]         u16 LE              nullifier_shard_prefix[0]
[..]         u16 LE              nullifier_shard_prefix[1]
[..]         [32]                relayer_fee_recipient
[..]         u32 LE              raw_adapter_ix_data.len()
[..]         raw_adapter_ix_data
[..]         u32 LE              action_payload.len()
[..]         action_payload
```

The variable-length `raw_adapter_ix_data` is what's forwarded to the
adapter program; `action_payload` is what the proof bound (the user's
proof of intent). They are typically identical for Jupiter (the
forwarded data IS the action payload), and slightly different for
Kamino (the forwarded data includes Kamino-specific Anchor framing
around the Borsh `KaminoAction`).
