---
title: "b402 SDK error codes and troubleshooting"
description: "Reference for all B402ErrorCode values thrown by the SDK, with causes, typical on-chain error messages, and steps to resolve each issue."
---

The SDK throws a `B402Error` for every recoverable failure. Each error carries a `code` from the `B402ErrorCode` enum, a human-readable `message`, and an optional `details` object with additional context. You can import both from `@b402ai/solana`:

```typescript
import { B402Error, B402ErrorCode } from '@b402ai/solana';

try {
  await b402.swap({ inMint: USDC, outMint: SOL, amount: 1_000_000n });
} catch (err) {
  if (err instanceof B402Error) {
    console.error(err.code, err.message);
  }
}
```

## Error codes

<AccordionGroup>

<Accordion title="INVALID_CONFIG — InvalidConfig">

**Enum value:** `B402ErrorCode.InvalidConfig`

**Causes:**

- `B402Solana` was constructed with both `keypair` and `signer` set. Pass one or the other.
- Neither `keypair` nor `signer` was supplied.
- `prover` or `proverArtifacts` was not provided and the zero-config circuit loader could not resolve artifacts (e.g., in a restricted network environment).
- `adaptProver` or `adaptProverArtifacts` was not provided when calling `privateSwap`, `lend`, or `redeem`.
- `privateSwap` was called without a `photonRpc` (Light Protocol Photon RPC client) — required for the nullifier non-inclusion proof.
- `privateSwap` was called without an `alt` (Address Lookup Table) PublicKey.
- The relayer URL returned a pool program ID that does not match the configured `programIds.b402Pool`.
- `b402.lend()` or `b402.redeem()` was called on devnet or localnet (Kamino mainnet-only).
- `private_swap` detected an exact-match constraint violation: you hold deposits in `inMint` but none with a value equal to `amount`.
- A note was passed explicitly to `privateSwap` whose `value` does not equal `amount`.

**Resolution:**

- Pass exactly one of `keypair` or `signer` to the constructor.
- Pass `proverArtifacts` (path to wasm + zkey) or let the SDK fetch them via the zero-config loader (requires network access on first run).
- For `privateSwap` calls, supply `photonRpc: createRpc(rpcEndpoint, rpcEndpoint)` from `@lightprotocol/stateless.js`.
- For the exact-match deposit error: shield an amount that exactly matches the swap amount you want to execute. Available deposit sizes are listed in the error message.

```
INVALID_CONFIG: private_swap requires an exact-match deposit.
Available deposit sizes for this mint: [500000, 2000000].
Requested: 1000000.
To swap a different amount, first shield exactly that amount,
or unshield and reshield to split.
```

</Accordion>

<Accordion title="NO_SPENDABLE_NOTES — NoSpendableNotes">

**Enum value:** `B402ErrorCode.NoSpendableNotes`

**Causes:**

- `unshield()` was called with a `mint` that has no spendable notes in the local `NoteStore`.
- `unshield()` was called with no `mint` and no prior `shield()` call in this session.
- `privateSwap()` was called and no notes exist in `inMint`.
- `b402.redeem()` was called but no voucher notes for the resolved collateral mint exist in the store.

**Resolution:**

- Call `b402.shield()` first to create a spendable note in the requested mint.
- If you have shielded on a different machine or in a prior session, call `b402.status({ refresh: true })` or `b402.balance({ refresh: true })` to re-scan on-chain history and populate the local store.
- To unshield a specific note that you know exists, pass it explicitly: `b402.unshield({ to, mint, note: myNote })`.

```typescript
// Re-sync from chain history before unshielding
await b402.status({ refresh: true });
await b402.unshield({ to: recipient, mint: USDC });
```

</Accordion>

<Accordion title="AMOUNT_OUT_OF_RANGE — AmountOutOfRange">

**Enum value:** `B402ErrorCode.AmountOutOfRange`

**Causes:**

- `amount` passed to `privateSwap()` was `<= 0n`.

**Resolution:**

- Pass a positive non-zero bigint for `amount`. All amounts are in smallest token units (e.g., `1_000_000n` for 1 USDC with 6 decimals).

</Accordion>

<Accordion title="RPC_ERROR — RpcError">

**Enum value:** `B402ErrorCode.RpcError`

**Causes:**

- A submitted transaction landed on-chain with a program error (the on-chain tx returned an `err` field).
- A transaction was not confirmed within the polling timeout (~6 seconds after submission for confirmation, 90 seconds for the initial poll on `privateSwap`).
- The on-chain tx failed post-simulation with an unexpected error.

**Resolution:**

- Check the transaction signature on Solscan or your cluster explorer. The transaction may have landed successfully even if the SDK threw — the poll timeout fires before finalization.
- If the transaction did land with an error, the error details from the Solana runtime are in `err.message`. Common on-chain errors: `Custom(2)` from the pool usually indicates a Merkle root mismatch (root expired) — retry with a fresh `shield()` or wait for the tree state to sync.
- The SDK does NOT mark a note as spent unless it confirms the transaction landed cleanly. A `RpcError` thrown after submission means your note is safe to retry.

<Warning>
  If you see `RpcError` and you are unsure whether the transaction landed, check Solscan before retrying. Retrying a swap with a note whose nullifier is already on-chain will fail at the on-chain nullifier check — it will not double-spend, but it will cost a failed transaction.
</Warning>

</Accordion>

<Accordion title="INVALID_SEED — InvalidSeed">

**Enum value:** `B402ErrorCode.InvalidSeed`

**Causes:**

- The wallet seed derivation failed. This can occur if a custom `B402Signer` returns an unexpected seed format from `getSeed()`.

**Resolution:**

- Verify that your `B402Signer` implementation returns a 32-byte `Uint8Array` from `getSeed()`. For `WalletAdapterSigner`, ensure the connected wallet supports `signMessage`.

</Accordion>

<Accordion title="PROOF_GEN_FAILED — ProofGenerationFailed">

**Enum value:** `B402ErrorCode.ProofGenerationFailed`

**Causes:**

- The Groth16 proof generation (via snarkjs) failed. Usually caused by an invalid witness — e.g., a note value of 0, an incompatible `proverArtifacts` version, or a corrupted wasm/zkey file.

**Resolution:**

- Confirm the circuit artifacts match the deployed verifier version. Check the artifact SHA256 against the pinned `circuits-v1` release tag on the b402 GitHub repository.
- Verify that `amount > 0` and that the note's `value` matches `amount` for `privateSwap`.

</Accordion>

<Accordion title="RELAYER_UNREACHABLE — RelayerUnreachable">

**Enum value:** `B402ErrorCode.RelayerUnreachable`

**Causes:**

- The configured `relayerHttpUrl` did not respond or returned an error on the `/health` check in `ready()`.

**Resolution:**

- Check that `relayerHttpUrl` is reachable from your environment. The hosted mainnet relayer is at `https://b402-solana-relayer-mainnet.run.app`.
- If the hosted relayer is unavailable, pass `relayer: yourKeypair` to the constructor to self-pay gas temporarily.

</Accordion>

<Accordion title="SLIPPAGE_EXCEEDED — SlippageExceeded">

**Enum value:** `B402ErrorCode.SlippageExceeded`

**Causes:**

- The on-chain swap's actual output was below the proof-bound `expectedOut` floor (i.e., the pool rejected the tx because `actualOut < expectedOut`).

**Resolution:**

- Increase `slippageBps` in the `swap()` call.
- Fetch a fresh `quoteSwap()` immediately before calling `swap()` and pass `expectedOut` from the updated quote.

</Accordion>

<Accordion title="TX_TIMEOUT — TxTimeout">

**Enum value:** `B402ErrorCode.TxTimeout`

**Causes:**

- The transaction was submitted but not confirmed within the configured timeout window.

**Resolution:**

- Transaction confirmation can lag on busy networks. Check Solscan before retrying — the transaction may have landed after the SDK's poll window closed.
- Use a private RPC endpoint (Helius, Triton) for more reliable submission and confirmation.
- Notes are never marked spent on a timeout, so retrying is safe.

</Accordion>

<Accordion title="NULLIFIER_SPENT — NullifierAlreadySpent">

**Enum value:** `B402ErrorCode.NullifierAlreadySpent`

**Causes:**

- The note you are trying to spend has already been spent in a previous transaction. This can happen when running two SDK instances with the same keypair concurrently, or after importing a note that was spent in a prior session.

**Resolution:**

- Call `b402.status({ refresh: true })` to re-sync the local note store from on-chain history. Spent notes will be filtered out and the store will reflect current spendable notes only.

</Accordion>

<Accordion title="INSUFFICIENT_BALANCE — InsufficientBalance">

**Enum value:** `B402ErrorCode.InsufficientBalance`

**Causes:**

- The wallet does not have enough SOL or SPL tokens to pay for a `shield` transaction or a setup transaction (e.g., creating a Kamino obligation).

**Resolution:**

- Ensure the wallet has enough SOL for transaction fees (~5,000 lamports for `shield`) and any one-time rent fees (Kamino first-lend costs ~0.04 SOL).
- For devnet testing, run `solana airdrop 1 --url devnet` to fund the wallet.

</Accordion>

<Accordion title="NOT_IMPLEMENTED — NotImplemented">

**Enum value:** `B402ErrorCode.NotImplemented`

**Causes:**

- A code path or adapter feature is scaffolded but not yet live on the target cluster.

**Resolution:**

- Check the adapter status table in the [program IDs reference](/reference/program-ids) for current availability. Features marked "scaffold" or "spec only" are not yet callable.

</Accordion>

</AccordionGroup>

## Common issues

### "exact-match deposit required"

`private_swap` and the underlying `privateSwap()` require that you hold a shielded note whose value is exactly equal to the `amount` you want to swap. The circuit currently spends the entire note — partial spends with change returned to the input mint are not supported in the current wire format.

**Fix:** shield the exact amount you intend to swap, then call `swap`.

```typescript
await b402.shield({ mint: USDC, amount: 1_000_000n }); // exact 1 USDC
await b402.swap({ inMint: USDC, outMint: SOL, amount: 1_000_000n });
```

If you have a larger deposit and want to swap a portion, unshield the large note back to your wallet, then re-shield the exact amount you need.

---

### "no note to unshield"

`unshield()` was called with no spendable note in the note store for the requested mint.

**Fix:** call `shield()` first, or if you shielded in a prior session, call `b402.status({ refresh: true })` to re-populate the local store from on-chain history.

---

### Kamino one-time rent (~$7 on first lend)

The first `b402.lend()` call for a given (keypair, mint) pair creates a Kamino `UserMetadata` and `Obligation` account, which costs approximately 0.04 SOL in account rent at current SOL prices.

This is a one-time cost per (keypair, mint) pair, not per lend. Subsequent lends in the same reserve cost only relayer transaction fees. The rent is refundable via Kamino's `close_obligation` instruction when the position is empty.

---

### Jupiter RouteNotFound on devnet

Jupiter only indexes mainnet liquidity. The `quote_swap` MCP tool and `b402.quoteSwap()` will return `RouteNotFound` when called on devnet.

**Fix:** switch to mainnet (`B402_CLUSTER=mainnet`) for live swap routing. On devnet, `private_swap` falls through to the mock adapter (constant 2x output) for protocol-level testing without real liquidity.

---

### Transaction size exceeds 1,232 bytes

Solana v0 transactions are capped at 1,232 bytes. The `adapt_execute` instruction is the heaviest flow and relies on the b402 Address Lookup Table (`9FPYufa1KDkrn1VgfjkR7R667hbnTA7CNtmy38QcsuNj`) to compress account metas.

**Fix:** use the default ALT. Do not pass custom `alts` without also including the b402 ALT — the SDK constructs the ALT list starting from the main b402 ALT and appending any extras you provide.

If you are building a custom adapter with many remaining accounts, extend the b402 ALT with your adapter's frequently-used accounts. Account meta count — not instruction count — is the binding constraint.

---

### Root expired (on-chain Custom error)

Merkle roots have a limited validity window on-chain. If you generate a proof against a root that was superseded by the time your transaction lands, the verifier rejects it.

**Fix:** retry the operation. The SDK fetches a fresh tree state immediately before proving, so a retry should succeed unless there is sustained high-frequency shielding activity. The indexer-backed path (`indexerUrl` configured) verifies the returned root against on-chain state before using it, which closes the window further.
