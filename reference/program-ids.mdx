## Build provenance

Mainnet binaries are built with explicit Cargo features. Use the same
feature string when reproducing bytecode from a tagged SHA.

[`scripts/build-all.sh`](https://github.com/mmchougule/b402-solana/blob/main/scripts/build-all.sh)
is a convenience wrapper that runs `cargo build-sbf` **without**
`--features` — fine for local `.so` iteration, but **not** a faithful
reproduction of what shipped on mainnet. After checkout, run
`cargo build-sbf --tools-version v1.54 --manifest-path
programs/<crate>/Cargo.toml --features '<comma-separated>'` using the
table below (same ordering as
[`ops/prd-35-mainnet-deploy.sh`](https://github.com/mmchougule/b402-solana/blob/main/ops/prd-35-mainnet-deploy.sh)
for verifier-adapt + pool, and
[`README.md`](https://github.com/mmchougule/b402-solana/blob/main/README.md)
for Kamino `per_user_obligation,cpi-only`).

| Program | Features |
|---|---|
| `b402-pool` | `prd_35_pending_inputs,inline_cpi_nullifier,phase_9_dual_note` |
| `b402-verifier-adapt` | `phase_9_dual_note` |
| `b402-nullifier` | `cpi-only` |
| `b402-kamino-adapter` | `per_user_obligation,cpi-only` |
| `b402-verifier-transact` | (none) |
| `b402-jupiter-adapter` | (none) |

Mismatched features = mismatched wire format. Borsh deserialization
in the SDK will reject misaligned binaries (Anchor error code 102 /
`InstructionDidNotDeserialize`).

The `cpi-only` feature on `b402-nullifier` is enforced by
[`programs/b402-nullifier/src/lib.rs:96-114`](https://github.com/mmchougule/b402-solana/blob/main/programs/b402-nullifier/src/lib.rs#L96-L114):
the program rejects any direct invocation and any CPI whose top-level
caller is not `b402_pool` (`42a3hsCXt…`). Nullifier address derivation
mixes `SEED_NULL = b"b402/v1/null"` with the program ID, matching
`DomainTags.nullifier` in
[`packages/shared/src/constants.ts:18`](https://github.com/mmchougule/b402-solana/blob/main/packages/shared/src/constants.ts#L18).

## Address Lookup Table (ALT)

The SDK uses an ALT to compress stable high-frequency accounts
(programs, protocol PDAs, common mints, adapter scratch ATAs) from
32 B to 1 B per reference inside `adapt_execute` txs — required to
fit Jupiter routes under Solana's 1,232 B tx-size cap (PRD-04 §5.2).

| Cluster | ALT address |
|---|---|
| mainnet | `3TSPLsa8aM5Xg9n8EHMuV5SK85RuMP96veFjv4BVrK9f` |
| devnet | `9FPYufa1KDkrn1VgfjkR7R667hbnTA7CNtmy38QcsuNj` |

Constants live in
[`packages/shared/src/constants.ts`](https://github.com/mmchougule/b402-solana/blob/main/packages/shared/src/constants.ts)
as `B402_ALT_MAINNET` and `B402_ALT_DEVNET`. SDK clients pick the
right one automatically from the cluster passed to `B402Solana`.

all programs at deterministic IDs derived from keypairs in
`ops/keypairs/` (gitignored). Localnet IDs differ per operator;
reference them via `Anchor.toml` rather than hard-coding. The
`[programs.localnet]` block currently pins:

## Verifying a deployed binary against source

```bash
# 1. Build the program for SBF at the same toolchain version as the
#    mainnet binary.
cargo build-sbf --tools-version v1.54 \
  --manifest-path programs/b402-pool/Cargo.toml \
  --features prd_35_pending_inputs,inline_cpi_nullifier,phase_9_dual_note

sha256sum target/deploy/b402_pool.so

# 2. Pull the on-chain program data and hash it.
solana -u mainnet-beta program dump \
  42a3hsCXtQLWonyxWZosaaCJCweYYKMrvNd25p1Jrt2y \
  /tmp/onchain.so
sha256sum /tmp/onchain.so
```

Hashes match modulo Anchor's metadata trailer. Reproducible-build
flags (e.g. `cargo-build-bpf --verifiable`) get you bit-for-bit
matching when available.

## Migration policy

When the pool's program ID changes, the migration policy from
[`PRD-01 §4.4`](https://github.com/mmchougule/b402-solana/blob/main/docs/prds/PRD-01-architecture.md#44-migration-policy)
applies:

1. Old pool stays operational so users can always exit.
2. New pool deploys with new program ID + new verifier.
3. SDK exposes a `migrationUnshield` flow that proves ownership in
   the old pool and shields into the new pool atomically.
4. Migration is opt-in.
