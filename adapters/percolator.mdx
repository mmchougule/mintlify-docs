`b402-percolator-adapter` integrates [percolator-prog](https://github.com/aeyakovenko/percolator-prog),
a permissionless perp engine on Solana. Positions land on percolator's
slab account but are **owned by a PDA derived from the user's private
spending key, not their wallet** — so an observer reading the slab
cannot map slots back to wallets without the spending keys.

Source: [`programs/b402-percolator-adapter/`](https://github.com/mmchougule/b402-solana-ex/tree/main/programs/b402-percolator-adapter).
End-to-end writeup: [`docs/findings/percolator-e2e.md`](https://github.com/mmchougule/b402-solana-ex/blob/main/docs/findings/percolator-e2e.md).
Spec: PRD-36 (Option B — shielded user / private positions).

## The owner_pda story

`slab.accounts[user_idx].owner` is a PDA derived from the user's
private spending key, not their wallet:

```
owner = PDA(["b402/v1", "perp-owner", LE32(spendingPub)], adapter_program_id)
```

From [`programs/b402-percolator-adapter/src/pda.rs:43-51`](https://github.com/mmchougule/b402-solana-ex/blob/main/programs/b402-percolator-adapter/src/pda.rs#L43-L51).
For the devnet keystone tx (alice signing `77zbXcUG…`):
`slab.accounts[1].owner == 4K8dopXjhc8qtBTQEEYAbneaZWJMhKqCg3HQAinwK4Pv`.
Different spending keys → different PDAs → different slab slots, by
construction.

Cross-adapter scoping holds by construction: the same viewing key,
used against `b402-kamino-adapter` vs `b402-percolator-adapter`,
yields different `owner_pda`s because the program ID is part of the
PDA seeds. b402's adapters cannot collide identities across
protocols.

## Per-slab `perp_mapping` bootstrap

The adapter holds a per-slab account (`perp_mapping`) that maps
`viewing_pub_hash → user_idx`. The mapping table is sized at
**`PERP_MAPPING_ACCOUNT_LEN = 81,968 B`** to hold the full per-slab
user table (well above Solana's 10,240 B per-instruction realloc cap).

Bootstrap is split into two ixs:

| Ix | Effect |
|---|---|
| `init_mapping` | Allocates the PDA at `MAX_PERMITTED_DATA_INCREASE = 10,240 B` and funds it with rent for the FULL target. |
| `grow_mapping` × 8 | Reallocates +10,240 B per ix until the account reaches `81,968 B`. Idempotent — no-op once at full size. |

All 9 ixs ride in a single transaction via the SDK helper. The cap is
[`MAX_PERMITTED_DATA_INCREASE`](https://github.com/mmchougule/b402-solana-ex/blob/main/programs/b402-percolator-adapter/src/lib.rs#L142),
not an arbitrary number — it's the max account-data growth the Solana
runtime allows in a single ix.

## The cpi-only security gate

Without the gate, an attacker could call `execute(ClosePosition)`
directly with another user's `viewing_pub_hash` (which is public in
the prior open's action_payload) and the adapter would happily sign
percolator's `WithdrawCollateral` as that user's `owner_pda`. With
`out_vault` having no owner constraint in the `Execute<'info>`
struct, the attacker passes their own USDC ATA and walks off with
the user's collateral.

The `cpi-only` Cargo feature enforces two checks at the top of
`execute`:

```rust
require!(get_stack_height() > 1, PercolatorAdapterError::DirectCallRejected);
let outer_ix = load_instruction_at_checked(current_idx, ix_sysvar)?;
require!(
    outer_ix.program_id == B402_POOL_PROGRAM_ID,
    PercolatorAdapterError::CallerNotB402Pool
);
```

From [`programs/b402-percolator-adapter/src/lib.rs:89-106`](https://github.com/mmchougule/b402-solana-ex/blob/main/programs/b402-percolator-adapter/src/lib.rs#L89-L106).
Mirrors the kamino-adapter cpi-only check (same threat model — pool
must be the only legal caller).

## Wire issues caught against percolator-prog

Six layout / wire issues surfaced during integration, all fixed against
percolator-prog `origin/main` (commit `04b854e`, engine pin
`5059332f8a`). All six are integration-side, not bugs in percolator's
logic:

1. **Slab `MAGIC` byte-order** — percolator stores `0x504552434f4c4154`
   in native LE; bytes-on-disk read `"TALOCREP"` (BE form of `"PERCOLAT"`).
2. **`Account` struct layout** — Solana BPF target is 8-byte i128
   alignment vs 16-byte host. On-chain `Account` is 416 B with
   `owner` at +184, `position_basis_q` at +64, `capital` at +0.
3. **Mapping PDA > 10,240 B** — the `init_mapping` + `grow_mapping × 8`
   split (above).
4. **SPL transfer ordering** — `InitUser` charges `fee_payment_if_init`
   from the user's slab ATA *before* the rest of the open path runs;
   the adapter must transfer adapter_in_ta → user ATA first, then
   InitUser, then DepositCollateral with the post-fee remainder.
5. **Engine `f6b13f57` pin** — `Account` struct grew +64 B and
   `MarketConfig` grew +56 B between `a946e550` and `f6b13f57`.
6. **Hyperp `MAX_ACCRUAL_DT_SLOTS = 10`** — Hyperp markets need a
   fresh `KeeperCrank` within ~10 slots of every trade or the engine
   returns `CatchupRequired`. Mainnet has keeper bots; devnet test
   harness pushes a fresh mark + crank before each open.

Full notes: [`docs/findings/percolator-e2e.md#bugs-found-integrating-against-percolator-prog`](https://github.com/mmchougule/b402-solana-ex/blob/main/docs/findings/percolator-e2e.md#bugs-found-integrating-against-percolator-prog).

## Reproduction

### Local mainnet-fork (open + close round trip)

```bash
tests/v2/scripts/start-percolator-fork.sh
pnpm exec vitest run tests/v2/e2e/v2_fork_percolator.test.ts
```

Expected ladder:

```
T1 — pool initialized + percolator adapter registered + token_config added
T2 — percolator market bootstrapped (slab + LP + matcher)
T3 — perp_mapping PDA at full PERP_MAPPING_ACCOUNT_LEN (=81968 B)
T4 — user b402 wallet shielded a 5 USDC (test mint) note via pool.shield
T5 — privatePerpOpen lands position on slab; owner = owner_pda(spendingPub)
```
